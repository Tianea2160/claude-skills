# 설계 철학

이 문서는 claude-skills의 각 스킬이 **왜 이런 구조로 설계되었는지**를 설명한다.

---

## 핵심 원칙: 컨텍스트 엔지니어링

같은 AI 모델이라도 **어떤 컨텍스트를 제공하느냐**에 따라 결과의 질이 극적으로 달라진다. 이 프로젝트의 모든 스킬은 Claude Code에게 **최적의 컨텍스트를 체계적으로 전달**하는 것을 목표로 한다.

> 프롬프트는 코드다. 버전 관리하고, 리뷰하고, 테스트해야 한다.

---

## claude-md-writer

### 문제 인식

CLAUDE.md는 Claude Code의 행동을 결정하는 핵심 파일이지만, 많은 프로젝트에서 다음과 같은 문제가 발생한다:

| 문제 | 결과 |
|------|------|
| 너무 길다 (200줄 초과) | Claude의 지시 준수율이 하락한다 |
| 코드에서 알 수 있는 내용을 반복한다 | 토큰을 낭비하고 핵심이 묻힌다 |
| 실행 불가능한 명령어가 있다 | Claude가 빌드/테스트에 실패한다 |
| 일반적인 best practice를 나열한다 | Claude가 이미 아는 것을 반복하여 가치가 없다 |

### 설계 근거

**1. 200줄 제한**

Anthropic 공식 문서에서 명시한 권장사항이다. 파일이 길어질수록 Claude가 뒤쪽 지시를 무시할 확률이 높아진다. 200줄은 핵심만 담기에 충분하면서도 Claude의 주의력이 유지되는 임계점이다.

**2. "이걸 빼면 Claude가 실수하나?" 테스트**

매 줄마다 이 질문을 적용한다. 답이 "아니오"면 그 줄은 삭제한다. 이 원칙은 Anthropic이 제시한 핵심 필터링 기준이다.

**3. 계층적 분리 (CLAUDE.md → rules → skills)**

Toss 테크블로그에서 제시한 3-Tier 지식 아키텍처에서 영감을 받았다:

```
Global  (~/.claude/CLAUDE.md)     — 회사/팀 공통 표준
Domain  (.claude/rules/)          — 특정 경로에만 적용되는 규칙
Local   (CLAUDE.md)               — 저장소별 구현 맥락
```

모든 규칙을 하나의 CLAUDE.md에 넣으면 토큰이 낭비된다. path-scoped rules를 사용하면 **해당 파일을 작업할 때만** 관련 규칙이 로드되어 토큰 효율이 높아진다.

**4. 5단계 워크플로우 (Discovery → Analysis → Interview → Generate → Validate)**

좋은 CLAUDE.md를 작성하려면 코드베이스를 이해해야 한다. 하지만 코드만으로는 충분하지 않다 — 팀의 워크플로우, 아키텍처 결정의 이유(WHY), gotcha 등은 사람에게 물어야 한다. 이 워크플로우는 **코드 분석**과 **사용자 인터뷰**를 모두 포함하여 완전한 CLAUDE.md를 생성한다.

**5. 참조 문서 분리 (references/)**

best-practices.md와 section-guide.md를 별도 파일로 분리한 이유는:
- SKILL.md 자체의 길이를 관리하기 위해
- 참조 자료를 독립적으로 업데이트할 수 있게 하기 위해
- Claude가 필요할 때만 참조하도록 하기 위해

## new-feature

### 문제 인식

AI에게 "기능 만들어줘"라고 하면 흔히 발생하는 문제들:

| 문제 | 원인 |
|------|------|
| 요구사항과 다른 결과물 | 요구사항을 충분히 확인하지 않음 |
| 기존 코드와 불일치 | 코드베이스를 분석하지 않음 |
| 과도한 엔지니어링 | 범위를 제한하지 않음 |
| 버그가 있는 코드 | 검토 단계가 없음 |

### 설계 근거

**1. 4단계 강제 워크플로우**

```
리서치 → 작업 → 검토 → 보고
```

각 단계를 건너뛸 수 없게 설계했다. 특히 **리서치 → 작업 사이의 승인 게이트**가 핵심이다. Claude가 코드를 작성하기 전에 반드시 사용자의 승인을 받아야 한다. 이를 통해:
- 잘못된 방향으로 대량의 코드를 작성하는 것을 방지
- 사용자가 접근 방식을 사전에 검토/수정할 기회 제공
- 재사용 가능한 기존 코드를 먼저 찾아 중복 구현 방지

**2. 코드베이스 분석 우선**

새 코드를 작성하기 전에 기존 구현을 반드시 먼저 찾는다. 이 원칙은:
- 코드 중복을 방지한다
- 프로젝트의 기존 패턴과 일관성을 유지한다
- 불필요한 의존성 추가를 막는다

**3. 외부 조사 (WebSearch/WebFetch)**

업계 best practice를 조사하는 단계를 포함한 이유는:
- Claude의 학습 데이터에 최신 정보가 없을 수 있다
- 공식 문서의 권장 패턴을 따르는 것이 장기적으로 유지보수에 유리하다
- 알려진 함정을 사전에 파악하여 실수를 줄인다

**4. Task 기반 작업 관리**

TaskCreate/TaskUpdate를 사용하는 이유는:
- 대규모 기능을 검증 가능한 작은 단위로 분해
- 의존성을 명시하여 올바른 순서로 구현
- 진행 상황을 사용자에게 투명하게 공유
- 실패 시 어느 단계에서 문제가 발생했는지 특정 가능

**5. 자체 검토 단계**

Phase 3(검토)에서 코드를 다시 전수 검토하는 이유는:
- 구현 중에는 세부사항에 집중하느라 전체 그림을 놓칠 수 있다
- 엣지 케이스와 보안 취약점은 별도 검토 시 더 잘 발견된다
- 요구사항 대비 결과물을 체계적으로 대조하면 누락을 방지한다
