# New Feature Workflow

기능 개발을 4단계(리서치 → 작업 → 검토 → 보고)로 체계적으로 수행한다.

`$ARGUMENTS`가 기능 설명이다. 인자가 없으면 AskUserQuestion으로 목표를 먼저 수집한다.

---

## Phase 1: 리서치

**목표:** 무엇을 어떻게 만들지 명확히 정의한다.

### 1.1 요구사항 수집

AskUserQuestion으로 다음을 명확히 한다:
- **목표**: 구현하려는 기능이 정확히 무엇인가?
- **제약조건**: 기술적 제약, 호환성 요구사항이 있는가?
- **우선순위**: 핵심 기능 vs nice-to-have 구분
- **기대 결과**: 완성 시 어떤 상태여야 하는가?

불명확한 부분이 있으면 가정하지 말고 반드시 질문한다.

### 1.2 코드베이스 분석

Explore 에이전트를 병렬로 실행하여 기존 코드를 분석한다:

1. **재사용 가능한 코드** — 기존 유틸리티, 패턴, 컴포넌트 중 활용할 수 있는 것
2. **영향 범위** — 변경이 미치는 파일/모듈 범위
3. **기존 패턴** — 프로젝트에서 유사 기능이 어떻게 구현되어 있는지

새 코드를 작성하기 전에 기존 구현을 반드시 먼저 찾는다.

### 1.3 외부 조사

WebSearch/WebFetch로 best practice를 조사한다:
- 동일/유사 기능의 업계 구현 사례
- 공식 문서의 권장 패턴
- 알려진 함정이나 주의사항

### 1.4 제안 및 승인

조사 결과를 사용자에게 제안한다:

```
## 리서치 결과

### 접근 방식
- [권장 접근 방식과 근거]

### 재사용할 기존 코드
- [파일:라인 — 설명]

### 외부 참고사항
- [조사에서 발견한 best practice]

### 예상 변경 범위
- [수정/생성할 파일 목록]
```

AskUserQuestion으로 승인/수정/거부를 받는다.
**승인받지 않으면 Phase 2로 진행하지 않는다.**

---

## Phase 2: 작업

**목표:** 승인된 접근 방식으로 체계적으로 구현한다.

### 2.1 작업 분해

TaskCreate로 작업을 세분화한다. 각 태스크는:
- **독립적으로 완료 가능한 단위**여야 한다
- **의존성이 명확**해야 한다 (선행 작업 명시)
- **검증 가능**해야 한다 (완료 조건 명시)

예시:
```
Task 1: [선행 없음] 도메인 모델 정의
Task 2: [Task 1 의존] Repository 구현
Task 3: [Task 2 의존] Service 로직 구현
Task 4: [Task 1 의존] API 엔드포인트 작성
Task 5: [Task 3, 4 의존] 통합 테스트
```

### 2.2 순차 실행

1. 의존성이 없는 태스크부터 시작한다
2. 각 태스크 시작 시 TaskUpdate로 `in_progress`로 변경한다
3. 완료 시 TaskUpdate로 `completed`로 변경한다
4. 다음 태스크의 선행 조건이 충족되었는지 확인 후 진행한다

### 2.3 작업 원칙

- 기존 코드/패턴을 최대한 재사용한다
- 과도한 추상화를 피한다 — 현재 필요한 만큼만
- 보안 취약점을 만들지 않는다 (OWASP Top 10)
- 변경 범위를 최소화한다 — 요청된 것만

---

## Phase 3: 검토

**목표:** 결과물이 요구사항에 부합하고 결함이 없는지 확인한다.

### 3.1 목표 정합성 검증

Phase 1에서 합의한 요구사항을 하나씩 대조한다:
- [ ] 핵심 기능이 모두 구현되었는가?
- [ ] 제약조건을 준수했는가?
- [ ] 기대 결과와 일치하는가?
- [ ] 원래 방향에서 벗어나지 않았는가?

### 3.2 코드 품질 검증

변경된 코드를 전수 검토한다:

**흐름 검증:**
- 정상 케이스(happy path)가 올바르게 동작하는가?
- 에러 케이스가 적절히 처리되는가?

**엣지 케이스:**
- null/empty 입력
- 동시성 문제
- 경계값 (0, max, overflow)
- 권한/인증 누락

**보안:**
- 입력 검증 누락
- SQL/Command injection 가능성
- 민감 데이터 노출

### 3.3 문제 수정

발견된 문제는 즉시 수정한다. 수정 후 다시 3.1-3.2를 반복한다.

---

## Phase 4: 보고

**목표:** 작업 결과를 사용자에게 명확하게 전달한다.

### 보고 형식

```
## 작업 완료 보고

### 구현 내용
- [핵심 변경사항 요약]

### 변경 파일
| 파일 | 변경 유형 | 설명 |
|------|---------|------|
| `path/file` | 생성/수정 | 설명 |

### 검증 결과
- [검토에서 확인/수정한 사항]

### 추가 필요 작업
- [있다면 후속 작업 안내]
- [없으면 "없음"]

### 테스트 방법
- [사용자가 직접 확인할 수 있는 방법]
```
